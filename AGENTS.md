# DFD-HR 实验与工程规范

## 1. 适用范围

本文件约束 DFD-HR 仓库内的代码修改、实验设计、Jupyter 验证、训练、评估、产物管理和 Git 操作。自动化代理与项目成员均应遵守。

优先级如下：

1. 当前任务的明确要求。
2. 本文件。
3. 仓库中的专项文档和既有实现惯例。

发现要求冲突、数据协议含糊或操作可能破坏现有结果时，先停止相关写操作，记录冲突并取得确认。

## 2. 核心原则

- 代码、配置、测试和小型文档进入 Git。
- 数据集、环境、缓存、权重、日志和训练产物留在 Git 之外。
- Notebook 用于检查、诊断和分析，长期训练交给独立终端会话。
- 每次实验绑定唯一运行编号、冻结配置、Git 提交和独立运行目录。
- 每个实验只验证一个主要变量。失败实验也要保留状态和结论。
- 训练集用于训练，验证集用于选模，跨域测试集只用于训练完成后的最终评估。
- 公开仓库只保存可公开、可复用、已匿名化的内容。

## 3. 工程质量与最小实现

### 3.1 最小实现

- 只实现当前需求和验证该需求所必需的部分。
- 优先复用现有接口、配置结构和工具函数。
- 不为假设中的未来需求提前增加框架、抽象层、兼容层或配置项。
- 一个直接函数可以清楚解决的问题，不引入新的类、注册器或依赖。
- 不顺手重构无关代码。必要的邻近修复必须与当前目标直接相关。
- 删除被本次修改取代且确认无调用方的旧路径，避免新旧实现长期并存。

### 3.2 可维护性

- 模块、函数和配置项保持单一职责。
- 名称表达业务含义，避免 `tmp2`、`new_final`、`misc` 等模糊命名。
- 控制流保持直接。优先早返回，减少深层嵌套和隐式状态。
- 公共行为必须有稳定入口，调用方不得依赖实现细节。
- 注释解释约束、原因和不直观的决策，不复述代码表面行为。
- 修改行为时同步更新测试、配置示例和相关文档。
- 对数据协议、指标、checkpoint 和恢复逻辑的修改必须增加回归验证。

### 3.3 文件组织

任何新增文件在创建前必须确定长期归属。禁止因操作方便将文件随意放在仓库根目录或现有目录中。

| 内容 | 位置 |
| --- | --- |
| 模型、训练和评估代码 | `training/` 下对应模块 |
| 运行配置 | `training/config/` 下对应类别 |
| 自动化测试 | `tests/` |
| 可复用项目文档 | `docs/` |
| Notebook 源文件 | `notebooks/`，仅在确有交互验证需求时创建 |
| 小型测试数据 | `tests/fixtures/`，并明确检查大小和许可 |
| 临时诊断文件 | Git 外的当前运行目录 |
| 训练日志、指标和权重 | Git 外的当前运行目录 |

根目录只保留项目入口和仓库级控制文件，例如 `README.md`、`AGENTS.md`、许可证、依赖声明和安装入口。新建顶层目录前必须确认现有目录无法合理承载该内容。

不要为了匹配理想目录结构批量移动现有文件。目录迁移应单独提交，并验证导入路径、脚本入口和文档链接。

## 4. 路径与存储边界

本仓库不记录个人路径和服务器拓扑。运行时使用以下环境变量：

```text
DFDHR_REPO_ROOT       Git 仓库根目录
DFDHR_PYTHON          实验环境中的 Python 解释器
DFDHR_DATA_ROOT       只读数据集根目录
DFDHR_RUNTIME_ROOT    当前节点的实验运行根目录
DFDHR_ARCHIVE_ROOT    长期归档根目录或受控远端目标
DFDHR_CACHE_ROOT      模型与依赖缓存根目录
```

要求：

- `DFDHR_REPO_ROOT` 只保存 Git 跟踪内容和小型工作文件。
- `DFDHR_DATA_ROOT` 视为只读。禁止写入缓存、索引、预测、日志或 checkpoint。
- `DFDHR_RUNTIME_ROOT` 保存活跃实验，每个 `RUN_ID` 使用独立目录。
- `DFDHR_ARCHIVE_ROOT` 保存需要长期保留且已经校验的实验产物。
- 缓存、环境和大型产物不得写入共享代码目录。
- 同名本地路径不代表不同计算节点拥有相同内容。运行前必须验证数据清单。

禁止向公开仓库提交：

- 真实服务器名、用户名、IP、端口和 SSH 配置。
- 用户目录大小、GPU 当前使用者、进程号和节点占用。
- 内部绝对路径、私有挂载点和存储拓扑。
- 凭据、令牌、私钥和带认证信息的 URL。
- 未经确认可公开的数据来源清单和内部审计报告。

示例和文档统一使用 `compute-node`、`user-01`、`local-dataset-root` 等占位符。

## 5. Git 工作流

### 5.1 分支职责

- `main`：已验证代码、稳定配置、测试和公开文档。
- `fix/*`：单一、明确的缺陷修复。
- `feat/*`：可复用能力。
- `exp/*`：研究假设或实验变量。
- `infra/*`：可复用且不含内部信息的运行基础设施。
- `docs/*`：纯文档修改。
- `release/*`：论文级复现版本冻结。

禁止直接在 `main` 上开发。新任务从最新 `origin/main` 创建单一职责分支。服务器审计、训练代码和实验配置不得混入同一个分支。

### 5.2 修改与提交

修改前执行：

```bash
git status --short --branch
git fetch origin
git switch -c <type>/<topic> origin/main
```

提交前执行：

```bash
git status --short
git diff --check
git diff -- <明确文件>
```

规则：

- 使用 `git add <明确文件>`，避免未经检查的 `git add .`。
- 每个提交只包含一个逻辑变化。
- 提交信息采用 `<type>(<scope>): <summary>`，或保持仓库既有的简洁英文风格。
- 不改写他人的未合并历史，不覆盖用户的未提交修改。
- 推送前重新运行与修改范围相称的验证。

## 6. 实验定义

### 6.1 运行编号

每个实验使用唯一 `RUN_ID`：

```text
<project>_<train-set>_<core-variable>_<YYYYMMDD>_<sequence>
```

同一编号不得复用。运行目录为：

```text
${DFDHR_RUNTIME_ROOT}/runs/${RUN_ID}/
```

### 6.2 实验必须记录的信息

每个正式实验至少记录：

- 研究问题、假设、基线和唯一变化。
- 成功标准和失败后可得出的结论。
- `RUN_ID`、分支、提交哈希和工作树状态。
- 冻结配置及其 SHA-256。
- 数据协议、数据清单版本或哈希。
- 初始权重来源及其 SHA-256。
- Python、依赖、PyTorch、CUDA 和 GPU 信息。
- 完整启动命令、随机种子和有效 batch size。
- 训练日志、结构化指标、best 与 last checkpoint。
- 最终状态、异常、结果摘要和归档校验。

状态统一使用：

```text
planned
preflight
smoke_passed
running
completed
failed
aborted
archived
```

## 7. 实验生命周期

### 7.1 问题定义

启动前写清：

1. 要验证什么。
2. 基线是什么。
3. 唯一改变的变量是什么。
4. 成功标准是什么。
5. 失败结果能排除什么。

随后创建实验分支、`RUN_ID` 和冻结配置。禁止在启动正式训练时临时手改未提交配置。

### 7.2 Preflight

必须检查：

- Git 工作树干净，当前提交已记录。
- `DFDHR_PYTHON` 指向预期解释器，依赖可导入。
- GPU 型号、数量和可用显存符合配置。
- 本地磁盘有足够余量，运行目录具有明确空间预算。
- 数据路径、JSON 元数据和划分可解析。
- 数据清单与实验记录一致。
- 初始权重能够严格加载，哈希已记录。
- 输出目录位于 `DFDHR_RUNTIME_ROOT`，不在数据目录和代码目录。

任何一项失败都不得启动正式训练。

### 7.3 两批次 Smoke Test

按顺序验证：

1. 模型实例化。
2. 两个 batch 的 forward、loss、backward 和 optimizer step。
3. loss 有限，无显存溢出。
4. 预期冻结参数无梯度，可训练模块有梯度。
5. checkpoint 能保存并严格重新加载。
6. 峰值显存和运行时间被记录。

Smoke Test 只验证运行链路，不产生研究结论。

### 7.4 Mini Run

使用固定小子集、固定随机种子和与正式训练一致的有效 batch size，运行少量 epoch。必须验证：

- loss 和验证指标能正常更新。
- best 与 last checkpoint 的语义正确。
- 从 last checkpoint 恢复后，epoch、optimizer 和 scheduler 状态连续。
- 日志、结构化指标和退出状态完整。

Mini Run 结果不得作为论文主结果。

### 7.5 正式训练

- 使用 `tmux`、调度器或等价的持久会话启动。
- 多卡训练使用当前 PyTorch 支持的 `torchrun` 入口。
- 日志写入当前 `RUN_ID` 目录，并保留启动器退出状态。
- 明确记录每卡 batch、进程数、梯度累积步数和有效 batch size。
- 定期检查剩余磁盘，但监控不得高频遍历共享目录。
- 默认只保留 best 和 last checkpoint。额外周期权重需要明确实验理由和空间预算。
- checkpoint 先写临时文件，再原子替换正式文件。

### 7.6 验证与最终评估

- 训练期间只使用明确的验证集选择 best checkpoint。
- 验证集缺失时直接失败，禁止回退到测试集。
- 跨域测试集在训练和选模结束后执行。
- 禁止根据跨域测试结果反复调参并继续报告其为无偏测试结果。
- 帧级与视频级指标必须标明聚合方法、样本数和数据划分。
- 评估结果必须关联 checkpoint 哈希、配置哈希和代码提交。

### 7.7 归档

长期保留的最小集合：

```text
manifest.yaml
config.resolved.yaml
command.sh
environment.txt
git.txt
metrics.jsonl
training.log
best checkpoint
last checkpoint
summary.md
checksums.sha256
```

使用支持断点续传和带宽限制的工具同步。同步完成后比较源端与目标端 SHA-256。只有校验一致且归档可读取时，才允许清理本地旧产物。

缓存、重复权重、可再生特征和临时文件不进入长期归档，除非它们是实验结论所必需的证据。

## 8. Jupyter 规范

Notebook 适用于：

- 环境、路径和依赖检查。
- checkpoint 严格加载。
- 数据协议审计。
- 单卡显存 Smoke Test。
- 小规模推理、绘图和错误定位。
- 只读监控结构化日志。

Notebook 不承载长期训练、多卡长期 DDP 或依赖浏览器存活的任务。

每个 Notebook 只解决一个问题，并满足：

- 第一单元记录解释器、工作目录、Git 提交和 CUDA 状态。
- 固定随机种子，不依赖隐藏变量和终端遗留环境变量。
- `Restart Kernel` 后 `Run All` 可从头执行。
- 源 Notebook 进入 Git，执行后的副本进入对应运行目录。
- 不嵌入权重、大数组、完整日志或大量图片。
- 执行失败时保留错误输出，不覆盖上一次成功产物。

只有任务确实需要交互验证时才创建 Notebook。可由普通脚本和测试完成的验证不增加 Notebook。

## 9. 验证要求

验证强度与风险匹配：

- 文档修改：检查链接、命令、占位符、敏感信息和 `git diff --check`。
- 配置修改：解析配置并验证关键字段、路径和默认值。
- 通用代码修改：运行相关单元测试和完整 CPU 测试集。
- 数据协议修改：使用小型 fixture 验证划分、采样和泄漏防护。
- 训练逻辑修改：完成 CPU 单测、单卡两批次 Smoke、必要时执行多卡短 Smoke。
- checkpoint 或恢复修改：验证保存、严格加载和恢复状态连续性。
- 指标修改：使用手工可计算的固定输入验证数值。

项目测试使用当前实验环境执行：

```bash
"${DFDHR_PYTHON}" -m unittest discover -s tests -v
```

不能运行 GPU 验证时，明确记录未验证项和原因。禁止用 CPU 单测结果代替 GPU 训练链路验证。

## 10. 完成标准

代码任务完成必须同时具备：

- 需求范围内的最小实现。
- 文件位于明确、可维护的位置。
- 相关测试和检查具有本次运行的通过证据。
- Git diff 只包含预期文件，无调试残留和大型产物。
- 文档、配置与实际行为一致。
- 未引入服务器或用户敏感信息。

实验完成必须同时具备：

- 唯一 `RUN_ID` 和明确研究问题。
- 干净、可定位的 Git 提交与冻结配置。
- 可核验的数据协议、环境和初始权重。
- 完整命令、日志、指标、best 和 last checkpoint。
- 成功或失败的明确结论。
- 已校验的归档或明确的保留策略。

单独存在 checkpoint、可运行但无法从头执行的 Notebook、或缺少提交与配置关联的训练结果，均不构成可复现实验。
